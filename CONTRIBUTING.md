# 직원용 홈페이지 수정 절차

이 저장소의 `main` 브랜치는 실제 홈페이지 기준입니다. 직원은 `main`을 직접 수정하지 않고, 작업별 브랜치와 Pull Request(PR)를 사용합니다.

## 가장 쉬운 작업 순서

1. 담당 작업 하나를 정합니다. 서로 다른 수정은 한 브랜치에 섞지 않습니다.
2. 최신 `main`에서 새 브랜치를 만듭니다.
   - 디자인팀: `design/작업이름`
   - 광고·마케팅팀: `marketing/작업이름`
   - 기술팀: `tech/작업이름`
3. GitHub에서 `src/pages`의 담당 카테고리 또는 `src/assets`를 수정한 뒤 변경 내용을 저장합니다. 주소별 `index.html` 폴더나 `dist`는 직접 만들지 않습니다.
4. 비밀번호, API 비밀 키, 고객·신청자 개인정보는 파일이나 화면 캡처에 넣지 않습니다.
5. Draft PR을 만들고 PR 양식의 확인 항목을 채웁니다.
6. Cloudflare 미리보기 주소에서 PC·모바일 화면과 링크를 확인합니다.
7. 담당 승인자가 확인한 뒤에만 PR을 `main`에 병합합니다.

## 어떤 수정이 어디에 해당하는가

- 배너·강사진·폼 선택지 같은 반복 운영 콘텐츠는 직원 로그인 후 관리자 화면에서 수정합니다.
- 페이지 배치, 색상, 새 섹션 등 디자인 구조 변경은 `design/*` 브랜치에서 진행합니다.
- 광고 추적 코드, 캠페인 랜딩 문구 등은 `marketing/*` 브랜치에서 진행합니다.
- 데이터베이스, 로그인, 보안, 배포 설정은 `tech/*` 브랜치에서 진행합니다.

## 문제가 생겼을 때

- 다른 사람과 같은 파일을 동시에 고쳤다는 충돌 메시지가 나오면 임의로 해결하지 말고 기술 담당자에게 요청합니다.
- 미리보기에서 데이터가 보이지 않는다면 권한에 맞는 정상 동작일 수 있습니다. 개인정보 메뉴는 디자인팀 계정에 표시되지 않습니다.
- 운영 사이트가 바뀌지 않는 것은 정상입니다. `main`에 승인·병합되기 전까지 기능 브랜치는 미리보기에서만 확인합니다.

자세한 배포·보안 절차는 `docs/SECURE_ADMIN_CLOUDFLARE_RUNBOOK.md`를 참고합니다.
